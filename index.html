Git이란 무엇인가요? 분산 버전 관리 시스템

Git은 코드(또는 파일)의 변경 이력을 기록하고 관리하는 분산 버전 관리 시스템(DVCS)입니다. 간단히 말해, “내 프로젝트의 시간을 되돌릴 수 있게 해 주는 도구”라고 생각할 수 있습니다.

Git을 사용하면 다음과 같은 일들을 손쉽게 할 수 있습니다.

  • 변경 이력 추적
    언제, 누가, 어떤 파일을 어떻게 수정했는지 커밋 단위로 확인할 수 있습니다.
  • 이전 상태로 되돌리기
    문제가 생기면 특정 커밋 기준으로 되돌리거나 일부 변경만 되돌리는 것이 가능합니다.
  • 안전한 실험 (브랜치)
    새로운 기능을 브랜치에서 마음껏 실험하고, 안정화된 뒤에 메인 코드와 합칠 수 있습니다.
  • 원격 저장소를 이용한 협업
    GitHub, GitLab 같은 서비스와 연동하면 팀원들과 동시에 작업하고, 코드를 리뷰하며, 백업까지 겸할 수 있습니다.

Git이 필요한 순간 실전 예시

  • 1. 과제·개인 프로젝트 버전 관리
    파일 이름을 main_final_v3_final_really.zip 이런 식으로 관리하던 방식을 깔끔하게 정리해 줍니다.
  • 2. 팀 프로젝트 협업
    같은 파일을 여러 명이 동시에 수정해도, 브랜치와 머지를 통해 충돌을 해결하며 함께 작업할 수 있습니다.
  • 3. 안전한 실험과 롤백
    새로운 기능을 시도해 봤다가 망해도, 안정적인 커밋으로 바로 돌아갈 수 있습니다.
Git을 쓰는 순간 “되돌릴 수 있다”는 심리적 안정감을 얻습니다.

Git 설치 방법 Windows / macOS / Linux

Git은 대부분의 운영체제에서 무료로 설치할 수 있습니다. 아래는 대표적인 설치 방법 요약입니다.

Windows 설치 파일 실행
macOS Homebrew 권장
Linux 패키지 매니저 사용

Windows Git for Windows

  1. 공식 사이트에서 설치 파일 다운로드:
    https://git-scm.com/ 접속 → Download for Windows 클릭
  2. 다운로드한 .exe 파일 실행 후 설치 마법사의 기본 옵션을 그대로 두고 진행합니다.
  3. 설치 후 Git Bash 또는 PowerShell을 열어 아래 명령으로 설치 여부를 확인합니다.
check 
git --version

macOS & Linux 패키지 매니저 활용

macOS 
# Homebrew가 설치되어 있다면
brew install git

# 설치 확인
git --version
Linux 
# Debian / Ubuntu 계열
sudo apt update
sudo apt install git

# Fedora / CentOS 등
sudo dnf install git

# 설치 확인
git --version

설치 후 꼭 하는 첫 설정 전역 사용자 정보

Git은 커밋을 남길 때마다 이름과 이메일을 기록합니다. 한 번만 설정해 두면 이후 모든 저장소에서 공통으로 사용됩니다.

config 
git config --global user.name "홍길동"
git config --global user.email "example@example.com"

# 현재 설정 값 확인
git config --list

Git 기본 사용법 필수 명령어 빠르게 훑기

지금부터는 실제로 프로젝트를 Git으로 관리할 때 가장 자주 쓰는 명령어들을 흐름대로 정리합니다.

1. 저장소 초기화 / 클론 init / clone

먼저, 작업 폴더에 Git 저장소를 만들거나 이미 존재하는 원격 저장소를 가져옵니다.

새 프로젝트 
# 현재 폴더를 Git 저장소로 만들기
git init
원격 복제 
# 기존 GitHub 저장소를 가져오기
git clone <원격-저장소-URL>

# 예: 
git clone https://github.com/user/repo.git

2. 상태 확인 & 스테이징 status / add

파일을 수정한 뒤에는 어떤 파일이 변경되었는지 확인하고, 커밋에 포함할 파일을 스테이징 영역에 올립니다.

status 
# 현재 상태(추적 여부, 변경 사항 등) 확인
git status
add 
# 특정 파일만 스테이징
git add index.html

# 변경된 모든 파일 스테이징
git add .

3. 커밋 남기기 & 로그 보기 commit / log

스테이징된 변경 사항을 하나의 이력 단위(커밋)로 남깁니다.

commit 
# 커밋 메시지와 함께 저장
git commit -m "작업 내용을 짧고 명확하게"
log 
# 자세한 로그
git log

# 한 줄로 간단하게
git log --oneline

4. 원격 저장소와 동기화 remote / push / pull

로컬 저장소를 GitHub 같은 원격 저장소와 연결해 두면, 인터넷이 되는 어디에서나 코드를 가져오고 푸시할 수 있습니다.

remote 
# 원격 저장소 추가 (보통 origin 이라는 이름 사용)
git remote add origin <원격-저장소-URL>
push / pull 
# 로컬 main 브랜치를 원격 main 으로 푸시
git push -u origin main

# 원격 변경 사항 가져오기
git pull

브랜치로 안전하게 실험하기 branch / switch / merge

브랜치(branch)는 코드의 독립적인 작업 줄기입니다. 주로 main 또는 master 브랜치를 “배포용 안정 버전”으로 두고, 새로운 기능이나 버그 수정을 별도 브랜치에서 진행합니다.

브랜치 기본 명령어 생성 / 이동 / 목록

branch 
# 브랜치 목록 보기
git branch

# 새 브랜치 만들기 (현재 브랜치 기준)
git branch feature/login

# 브랜치로 이동 (Git 2.23+)
git switch feature/login

# 브랜치 생성 + 이동을 한 번에
git switch -c feature/signup

브랜치 합치기(머지) merge

작업을 마친 브랜치는 main 브랜치에 머지(merge)하여 반영합니다.

merge 
# 1. main 브랜치로 이동
git switch main

# 2. feature/login 브랜치를 main에 합치기
git merge feature/login

같은 부분을 서로 다르게 수정했다면 충돌(conflict)이 발생할 수 있습니다. 이 경우 Git이 표시해 주는 부분을 수동으로 수정한 뒤 다시 커밋하면 됩니다.

브랜치 개념을 그림으로 이해하기 시간축 상상 

main  o──o──o─────────────o
                       \         (배포용 안정 브랜치)
                        \
feature/login   o──o──o──o
                (로그인 기능 실험)
  • main에는 항상 “배포 가능한 상태”의 코드만 두고, 새로운 시도는 항상 브랜치에서 시작하는 습관을 들이면 좋습니다.
  • GitHub에서는 브랜치를 기반으로 Pull Request(PR)를 만들고, 코드 리뷰를 거쳐 안정적으로 머지하는 문화가 일반적입니다.

자주 쓰는 안전 장치 reset / restore (참고)

처음에는 과감한 되돌리기보다는, 새 브랜치를 만들어 안전하게 실험하는 방식을 추천합니다. 그래도 알아두면 좋은 명령어입니다.

작업 폴더 변경 취소(단일 파일) git restore 파일명
스테이징 취소 git restore --staged 파일명
마지막 커밋으로 되돌리기 git reset --hard HEAD
특정 커밋까지 되돌리기 git reset --hard 커밋ID

주의: --hard 옵션은 되돌린 이후의 변경 내용을 복구하기 어렵습니다. 정말 필요할 때만 사용하고, 일반적인 실험은 항상 별도의 브랜치에서 진행하는 것을 권장합니다.

Git 잘 쓰는 작은 팁들 습관이 실력

  • 커밋은 “기능 단위”로 작게 쪼개기 한 번에 너무 많은 변경을 묶기보다는, 의미 있는 단위로 나누어 커밋하면 나중에 이력을 읽고 디버깅하기 훨씬 편해집니다.
  • 커밋 메시지를 보고도 무슨 작업인지 이해되게 쓰기 예: fix: 로그인 실패 시 오류 메시지 한글화, feat: 회원 가입 폼 유효성 검사 추가 처럼 작성해 보세요.
  • 작업 시작 전에는 항상 git pull 팀원과 함께 작업 중이라면, 코딩을 시작하기 전에 원격 저장소의 최신 변경 사항을 받아오는 습관을 들이는 것이 좋습니다.
  • 작업이 애매하면 브랜치부터 파고 보기 “이거 해볼까 말까?” 싶은 실험은 가볍게 브랜치를 만들어 진행한 뒤, 마음에 안 들면 그냥 브랜치를 지워버리면 됩니다.
  • 처음에는 GUI 툴과 병행해도 괜찮습니다 SourceTree, GitHub Desktop 같은 GUI 도구를 쓰면서 핵심 명령어만 CLI로 익혀도 충분히 실전에서 활용 가능합니다.
